# 每次学习会话指南

这份文档定义每次说“继续学习”时的执行方式。目标是让学习按完成状态连续推进，而不是按日期硬跳。

## 会话开始

每次开始前先读取：

1. `README.md`
2. `docs/progress.md`
3. `docs/weekly-plan.md`
4. 必要时读取 `docs/code-interpreter-guard-design.md`
5. 当当前任务标注 AIRT 参考时，读取 `/home/work/airt` 中对应 lab 的 `README.md`

## 推进规则

- 以 `docs/progress.md` 的完成状态为准，日期只作为参考。
- 如果上次任务未完成，优先续上，不进入新任务。
- 每次只推进一个最小闭环。
- 每次正式学习必须留下可检查证据：代码、样例、日志、报告、笔记或失败记录至少一种。
- 使用 AIRT 时，每次最多抽取一个 lab 的一个攻击点，迁移成当前任务需要的受控样例。
- 周一到周五执行正式学习任务。
- 周六只做可选复盘、补漏、复跑实验和整理问题清单。
- 周日休息，不安排正式学习任务。
- 不擅自扩展到 MVP 外能力。
- 不把 AIRT 当成额外必刷课程；AIRT 的产出必须服务于 `Code Interpreter Guard` 的攻击样例、检测样例、报告或 Demo。

## 每次输出

每次学习会话应给出：

- 今日目标
- 核心概念解释
- 动手实验步骤
- 系统安全视角
- 如果使用 AIRT：说明选取的 lab、攻击点、迁移后的本地样例和 runtime 行为预期
- 验收标准
- 今日记录模板

## 会话结束

学习结束时更新：

- `docs/progress.md`
- `docs/notes/week-XX/` 下当天的技术文档（如 `day-01-first-call.md`）
- 当天的可检查证据：运行命令、实际输出、样例路径、日志/report 路径或失败记录

如果任务没有完成，记录：

- 当前做到哪里
- 阻塞点
- 下次继续的第一步

## 记录要求

每日笔记简短够用，建议 300-800 字。写作规范见 `docs/notes/README.md`。
