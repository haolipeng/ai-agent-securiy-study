## AIRT 课程使用方式

本计划把本地 `/home/work/airt` 的 AIRT 课程作为攻击侧素材库，用来构建可 replay 的 attack case suite；防御侧主线是 Agent Runtime Guard 产品雏形。

使用原则：

- 主线是攻防闭环：AIRT-derived attack case -> Agent tool use -> runtime behavior -> eBPF 观测 -> task 归因 -> policy detection / runner-layer enforce -> 报告。
- AIRT 只用于补攻击面理解、提供 payload 思路、生成受控 demo case 和回归输入。
- 每次最多抽取一个 AIRT lab 的一个攻击点，迁移成当前阶段需要的最小样例。
- 不以拿 flag 或完整刷完 AIRT 为目标，避免偏离 runtime security 主线。
- 所有从 AIRT 迁移的样例都必须转写成受控、本地、脱敏的 `examples/attacks/` 或 `examples/rag-min/` case。
- attack case 默认支持 `mock` 模式，直接读取固定 `expected_tool_call.json`；可选支持 `llm` 模式，用真实模型从 prompt 或恶意文档生成 tool call。
- 可以表达高危意图，但执行必须受控：模拟敏感文件放在 `lab/fixtures/secrets/`，外联打到本地测试服务或测试地址，破坏性命令只能作用于临时目录。

推荐映射：

| AIRT 内容 | 放入本计划的位置 | 迁移目标 |
|---|---|---|
| Lab 02 Prompt Injection | 第 3 周、第 5 周 | 诱导 Agent 调用 `run_shell_command` / `read_file` 的 tool abuse case |
| Lab 03 RAG Exploitation | 第 4 周、第 5 周 | 恶意文档注入后触发 tool call 的 RAG case |
| Lab 08 Full Engagement | 第 12-16 周 | 最终 Demo 攻击链和报告结构参考 |
| Lab 04 Multi-Agent Exploitation | 后续增强 | shared memory / delegation 污染导致工具滥用的攻击链参考 |
| Lab 07 Automation | 后续增强 | prompt injection / RAG 注入自动化测试参考 |

Attack case 标准结构：

```text
examples/attacks/<case_id>/

  input.txt 或 malicious_doc.txt
  expected_tool_call.json
  expected_runtime_events.json
```

## MVP 边界

必达闭环：

- Go 最小 Agent 支持 `run_shell_command`、`read_file`、`write_file`、`http_get`
- AIRT-derived attack case suite 支持 mock / llm 双模式，默认 mock 可离线 replay
- runner 记录 task / tool call 日志，并传递 `task_id`
- eBPF 采集 `execve`、`openat`、IPv4 `connect`
- Go aggregator 基于 runner pid、pid/ppid、进程树做 best-effort task 归因
- detector 输出敏感文件访问、危险命令、异常外联、敏感文件后外联报告
- runner-layer enforce 拒绝明显高危 tool call

可选事件增强：

- 采集 `unlink`
- 采集 `rename`

增强项，不作为 MVP 成败标准：

- 轻量行为基线
- eBPF 实时阻断或 `kill`
- `seccomp`、namespace、`cgroup`、容器隔离和容器归因

runner-layer enforce 属于 MVP：它只在 tool 执行前基于参数、路径、命令、URL 和策略拒绝明显高危行为，不承诺强隔离、内核级阻断或防逃逸。
